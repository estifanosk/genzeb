Load Testing Recommendation

Short Answer for Team

Load testing should be done in a dedicated production-like performance environment. Local testing is only useful for developer validation, not capacity or resilience. If a dedicated environment is not available, use staging with isolated Kafka topics, tables, consumer groups, and a scheduled test window. Production should only be used later for small, controlled shadow validation after lower-environment testing.

Recommendation

I recommend load testing this pipeline because it crosses Kafka, Java connectors, RDS, Schema Registry, tax and statement metadata queries, audit storage, DynamoDB, Kubernetes, and Datadog. Unit and integration tests will not prove whether the system can handle real volume, bursts, retries, lag, throttling, downstream slowness, or recovery after failures.

Benefits

  • Validate expected, peak, and burst throughput.
  • Measure end-to-end latency from input Kafka message to DynamoDB update.
  • Identify bottlenecks in RDS, DynamoDB, tax/statement queries, Kafka consumers, or Kubernetes resources.
  • Confirm whether Kafka consumer lag grows, stabilizes, or drains after bursts.
  • Validate retry behavior and make sure retries do not amplify incidents.
  • Confirm Datadog dashboards, alerts, logs, traces, and metrics work under realistic volume.
  • Verify pod CPU, memory, JVM heap, GC, thread pools, and database connection pools.
  • Understand capacity limits before production traffic exposes them.

Where to Test

The best option is a dedicated performance environment, such as perf, loadtest, or staging-perf.

The environment should use separate resources:

  • Kafka topics
  • Kafka consumer groups
  • Kubernetes namespace and deployments
  • RDS database or schema
  • DynamoDB tables
  • Schema Registry subjects if needed
  • Datadog tags such as env:perf and test_run_id

Do not run initial load tests in production. Production testing can create customer-impacting lag, overload RDS or DynamoDB, trigger false alerts, pollute audit tables, increase Datadog cost, or create bad downstream state.

What to Test

Kafka input topics
  -> source connector
  -> RDS staging tables
  -> events topic
  -> event processor
  -> tax / statement metadata queries
  -> entity change topic
  -> audit store
  -> sink connector
  -> DynamoDB

Also test major components individually when needed: Kafka input throughput, RDS writes, tax and statement queries, entity change publishing, audit writes, DynamoDB updates, and Datadog error visibility.

How to Test

  1. Define expected volume, peak volume, burst size, daily volume, acceptable lag, and acceptable end-to-end latency.
  2. Create representative Kafka test data across the four input topics.
  3. Include valid records, invalid schema records, missing-field records, duplicate records, large payloads, and records that trigger tax and statement queries.
  4. Use a custom Kafka load generator or Kafka performance tooling to publish messages at controlled rates.
  5. Run baseline, peak, burst, soak, failure, and recovery tests.
  6. Capture all behavior in Datadog dashboards and logs using env:perf and test_run_id.
  7. Document bottlenecks, capacity limits, tuning recommendations, and risk areas.

Recommended Test Types

  • Baseline test: expected production traffic.
  • Peak test: expected highest normal traffic.
  • Burst test: sudden spike for a short period.
  • Soak test: steady load for several hours.
  • Failure test: downstream slowdown or outage.
  • Recovery test: dependency returns and backlog drains.

Metrics to Track in Datadog

  • Kafka input and output rate
  • Kafka consumer lag by topic, partition, and consumer group
  • Processing latency per app
  • End-to-end latency
  • RDS write latency, errors, and connection pool usage
  • DynamoDB write latency, errors, and throttles
  • Tax and statement query latency and error rate
  • Retry count and failed-record count
  • Kubernetes CPU, memory, restarts, and scaling behavior
  • JVM heap, GC, and thread usage
  • Datadog log and metric ingestion volume

Failure Scenarios to Simulate

  • RDS slowdown or connection pool exhaustion
  • DynamoDB throttling
  • Tax or statement query timeout
  • Schema Registry unavailable
  • Kafka publish failure
  • Datadog unavailable, delayed, or rate-limited
  • Bad records mixed into normal traffic

Story Draft

Title: Load test Kafka-to-RDS-to-Dynamo pipeline

Goal: Validate that the pipeline can process expected, peak, and burst traffic while maintaining acceptable latency, lag, and error rates.

Scope: Define throughput targets, build or configure a Kafka load generator, generate realistic test messages, run baseline/peak/burst/soak/failure/recovery tests, capture Datadog metrics, identify bottlenecks, and document tuning recommendations.

Acceptance Criteria: The pipeline handles expected throughput without growing lag, handles peak throughput within the agreed latency target, absorbs burst traffic and drains backlog, validates retry and failure behavior, shows the full test behavior in Datadog, and documents bottlenecks and capacity recommendations.
